# Step 1. 쿠팡 연동 기준 확정

기준일: 2026-03-31  
상태: 완료  
목적: 구현 전에 "무엇을 기준으로 저장하고 구분할지"를 확정한다.

## 1. 이번 단계에서 확정된 내용

### 1.1 채널 구분

쿠팡은 네이버와 분리된 별도 채널로 취급한다.

- `source_channel = 'naver'`
- `source_channel = 'coupang'`

즉 동기화, 검색, 분석 모두 네이버와 쿠팡을 구분한다.

### 1.2 쿠팡 내부 배송유형 구분

쿠팡은 채널 하나로 끝나지 않는다.

같은 상품이라도 아래 두 유형을 구분해야 한다.

- 판매자배송: `marketplace`
- 로켓그로스: `rocket_growth`

즉 쿠팡은 아래 2축으로 본다.

- 채널: `coupang`
- 배송유형: `marketplace | rocket_growth`

### 1.3 상품 식별 기준

쿠팡의 상품군 ID와 실제 판매 SKU ID는 다르다.

- 상품군: `sellerProductId`
- 실제 판매 SKU: `vendorItemId`

이번 구현에서 실제 매핑 기준은 `vendorItemId`로 잡는다.

즉 쿠팡 연동의 외부 상품 식별자는 아래가 된다.

- `source_product_id = vendorItemId`

### 1.4 같은 옵션도 판매자배송/로켓그로스 SKU가 다를 수 있음

실제 API 확인 결과 같은 옵션이라도 아래처럼 ID가 다를 수 있었다.

예: 트릿백 `1개 민트`

- 일반판매 `vendorItemId = 94054315326`
- 로켓그로스 `vendorItemId = 94165163695`

즉 "같은 상품/같은 옵션"이어도 배송유형이 다르면 SKU ID가 다를 수 있다.

### 1.5 쿠팡 SKU 매핑 방식

`commerce_product_mappings`에는 SKU 1개당 1행으로 저장한다.

예시:

- `coupang + marketplace + 93885404452`
- `coupang + rocket_growth + 94132809744`

둘 다 내부적으로 같은 상품군 `엔자이츄`로 연결될 수 있다.

### 1.6 수량 묶음 상품은 pack multiplier가 필요함

쿠팡은 아래처럼 수량별 SKU가 별도로 존재한다.

- 엔자이츄 1개
- 엔자이츄 2개
- 엔자이츄 3개
- 엔자이츄 6개

따라서 SKU별로 `pack_multiplier`를 함께 관리해야 한다.

예:

- 엔자이츄 1개 -> multiplier 1
- 엔자이츄 2개 -> multiplier 2
- 엔자이츄 6개 -> multiplier 6

### 1.7 고객 분석은 초기엔 채널 분리 원칙

네이버는 `buyer_id` 기반 분석이 강하지만 쿠팡은 동일 수준의 안정적인 구매자 ID 구조가 약하다.

따라서 초기 원칙은 아래와 같다.

- 네이버 고객 분석과 쿠팡 고객 분석을 기본 분리
- 쿠팡은 쿠팡 내부 기준으로만 분석
- 교차채널 고객 통합은 이번 단계 범위에서 제외

## 2. 이번 단계에서 명시적으로 "아직 확정하지 않은 것"

아래 항목은 아직 구현 기준으로 고정하지 않는다.

### 2.1 주문 API에서 실제로 내려오는 line-level 식별 방식

아직 미확정:

- 판매자배송 주문 API가 어떤 식별 조합을 주는지
- 로켓그로스 주문 API가 어떤 식별 조합을 주는지
- `source_line_id`를 어떤 값으로 최종 고정할지

즉 이건 Step 2 검증 전까지 구현하지 않는다.

### 2.2 쿠팡 고객키 설계

아직 미확정:

- 쿠팡 고객 분석에서 쓸 안정 키
- 이름/안심번호/주소 조합 방식
- 네이버와 동일 집계 규칙 적용 가능 여부

즉 이건 Step 2 이후 별도 검증 전까지 구현하지 않는다.

### 2.3 광고비 데이터 수집 방식

아직 미확정:

- 광고비를 OpenAPI로 직접 받을 수 있는지
- 광고센터 보고서를 별도 수집해야 하는지

즉 광고 분석은 현재 범위에서 설계 고정하지 않는다.

## 3. 이번 단계 결과로 바뀐 작업 원칙

이전에는 "네이버 구조에 쿠팡 커넥터 추가" 수준으로 볼 수 있었지만,
이제는 아래처럼 원칙이 바뀌었다.

- 네이버와 쿠팡은 별도 동기화
- 쿠팡은 다시 판매자배송/로켓그로스 분리
- 상품 매핑 기준은 이름보다 SKU
- 쿠팡 상품 매핑은 `vendorItemId` 중심
- 검색/분석도 채널과 배송유형을 분리해야 함

## 4. Step 1 완료 판단

이번 단계는 아래 질문에 모두 답할 수 있게 되면 완료로 본다.

- 쿠팡은 네이버와 같은 구조인가?
- 아니오. 채널 내부에서도 배송유형이 갈린다.

- 쿠팡에서 상품 매핑 기준은 무엇인가?
- `vendorItemId`

- 같은 상품도 판매자배송/로켓그로스 ID가 다를 수 있는가?
- 그렇다.

- 고객 분석을 바로 네이버 방식으로 섞어도 되는가?
- 아니다. 초기엔 분리 원칙으로 간다.

이 기준은 확정으로 본다.
