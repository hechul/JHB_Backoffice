# PWA Phase 1 구현 계획

작성일: 2026-03-16
대상 서비스: JHBioFarm 백오피스
범위: 1단계 설치형 PWA만 적용

---

## 1. 목적

현재 백오피스는 웹 주소로 접속해 사용하는 구조다.  
Phase 1의 목적은 이 서비스를 다음 수준까지 올리는 것이다.

1. 모바일과 데스크톱에서 앱처럼 설치 가능하게 만든다.
2. 브라우저 탭이 아닌 독립 실행 형태(`standalone`)로 열리게 한다.
3. 기존 로그인, 권한, 업로드, 필터링, 근태, 자동화 기능을 깨지 않는다.
4. 캐시 때문에 기존 배포/업데이트 흐름이 꼬이지 않도록 매우 보수적으로 적용한다.

이 단계에서는 "앱처럼 설치"가 핵심이며, 오프라인 강화나 푸시 알림은 범위에서 제외한다.

---

## 2. 이번 단계에서 포함하는 것

### 2.1 설치형 PWA
- `manifest.json` 추가
- 앱 이름, 짧은 이름, 설명, 시작 경로 설정
- 아이콘 세트 등록
- `display: standalone` 적용
- 테마 컬러, 배경 컬러 지정

### 2.2 기본 서비스 워커
- 정적 에셋만 최소 캐시
- 앱 쉘 수준에서만 동작
- 로그인/권한/API 데이터는 캐시하지 않음

### 2.3 설치 UX
- 설치 가능 브라우저에서 홈 화면 추가/앱 설치 가능
- 데스크톱에서 Dock/앱 목록에 설치 가능

---

## 3. 이번 단계에서 하지 않는 것

다음 항목은 Phase 1 범위에서 제외한다.

- 푸시 알림
- 백그라운드 동기화
- 오프라인 사용성 강화
- API 응답 캐시
- 업로드 파일/필터링 결과의 오프라인 보존
- 복잡한 캐시 무효화
- 버전 업데이트 토스트

즉, 이번 단계는 "설치 가능"에 집중하고, 데이터는 계속 온라인 기준으로 처리한다.

---

## 4. 왜 보수적으로 가야 하는가

이 백오피스는 다음 특성이 있다.

- 로그인/권한이 중요함
- Supabase 세션 상태가 중요함
- 월/주차 필터, 고객 분석, 근태, 자동화처럼 최신 데이터가 중요함
- 배포 후 바로 최신 화면이 보여야 운영 혼란이 적음

따라서 PWA를 붙이더라도 API와 인증 흐름에 캐시를 적극적으로 걸면 안 된다.

Phase 1 원칙:

1. HTML 문서는 네트워크 우선
2. API 응답은 캐시하지 않음
3. 인증 관련 요청은 캐시하지 않음
4. 정적 번들(JS/CSS/아이콘)만 캐시

---

## 5. 구현 방향

## 5.1 파일 추가/수정 예상

예상 대상 파일:

- `nuxt.config.ts`
- `public/manifest.webmanifest`
- `public/icons/*`
- `app/app.vue` 또는 공통 head 설정 위치
- 서비스 워커 등록 파일
- 필요 시 PWA 관련 plugin/composable 1개

플러그인 선택지는 두 가지다.

### 선택지 A. Nuxt PWA 모듈 사용
- 장점: 설정이 빠름
- 단점: 캐시 기본값을 잘못 쓰면 운영상 위험

### 선택지 B. 서비스 워커를 직접 최소 구현
- 장점: 캐시 범위를 아주 명확하게 통제 가능
- 단점: 구현량이 조금 늘어남

권장:
- 이 프로젝트는 데이터와 인증이 민감하므로 **직접 최소 구현 또는 매우 보수적으로 설정한 모듈 사용**이 적합하다.

---

## 5.2 manifest 설계

기본안:

- `name`: `JHBioFarm 백오피스`
- `short_name`: `JHBioFarm`
- `start_url`: `/`
- `display`: `standalone`
- `background_color`: 현재 브랜드 배경 톤과 맞춤
- `theme_color`: 현재 헤더/포인트 컬러와 맞춤

아이콘:
- 최소 192x192, 512x512 필요
- maskable 아이콘도 가능하면 같이 준비

주의:
- 현재 로고 에셋 중 설치 아이콘으로 적합한 것을 골라야 함
- 배경이 투명한 로고만 쓰면 모바일 홈 화면에서 어색할 수 있음

---

## 5.3 서비스 워커 캐시 정책

Phase 1 캐시 전략:

### 캐시 대상
- 앱 번들 JS/CSS
- 폰트
- 아이콘
- 정적 이미지

### 캐시 제외
- Supabase 인증 요청
- `/api/*`
- 동적 페이지 데이터
- 업로드/다운로드 관련 요청
- 자동화 관련 요청

### 라우팅 기본 전략
- 문서/페이지 HTML: network-first
- 정적 리소스: stale-while-revalidate 또는 cache-first
- API: network-only

이 기준을 지키면:
- 설치는 되지만
- 데이터가 예전 상태로 남는 문제를 최소화할 수 있다.

---

## 6. 화면/UX 기대 결과

사용자 입장에서 Phase 1 이후 기대하는 변화:

1. 모바일 Safari/Chrome에서 홈 화면에 추가 가능
2. 데스크톱 Chrome/Edge에서 앱 설치 가능
3. 앱처럼 독립 창으로 열림
4. 기존 주소 접속 방식도 그대로 유지
5. 로그인/로그아웃/권한 동작은 기존과 동일

중요:
- 설치 UX는 추가되지만, 사용 흐름은 기존 웹과 같아야 한다.

---

## 7. 구현 시 주의사항

### 7.1 로그인/권한
- `useCurrentUser`, `auth.global.ts`, Supabase 세션 감지 흐름은 건드리지 않는다.
- 서비스 워커가 auth 관련 요청을 가로채지 않게 한다.

### 7.2 실시간/최신 데이터
- 대시보드
- 고객 분석
- 필터링
- 근태
- 자동화

위 화면은 "최신 데이터"가 중요하므로 캐시 개입을 최소화해야 한다.

### 7.3 배포/업데이트
- 현재는 GitHub push -> Vercel 자동 배포 구조다.
- PWA 적용 후에도 이 구조는 유지되어야 한다.
- 다만 서비스 워커가 낡은 번들을 계속 보여주지 않도록 캐시 버전 관리를 넣어야 한다.

### 7.4 모바일 테스트
- iPhone Safari
- Android Chrome
- 데스크톱 Chrome

최소 이 세 환경은 설치 여부 확인이 필요하다.

## 8. 권장 구현 순서

1. 아이콘 정리
2. `manifest.webmanifest` 추가
3. `nuxt.config.ts`에 manifest/head 연결
4. 최소 서비스 워커 적용
5. 설치 가능 여부 확인
6. 캐시가 로그인/권한/API에 영향 없는지 점검
7. 배포 후 설치/실행 smoke test

---

## 9. 테스트 체크리스트

### 9.1 로컬
- `npm run dev`에서 정상 렌더링
- 로그인/로그아웃 정상
- 고객 분석/대시보드/근태/자동화 정상
- manifest 로드 확인
- 서비스 워커 등록 확인

### 9.2 배포
- Vercel 배포 후 브라우저 설치 가능 확인
- 설치 후 앱 실행 확인
- 새 배포 후 기존 기능 이상 없는지 확인

### 9.3 회귀 테스트
- 로그인
- 회원가입
- 대시보드 조회
- 고객 분석 필터
- 업로드
- 필터링 실행
- 근태 출퇴근
- 자동화 진입

이 기능들은 PWA 적용 때문에 달라지면 안 된다.

## 10. 완료 기준

다음이 모두 만족되면 Phase 1 완료로 본다.

1. `manifest`와 서비스 워커가 정상 등록됨
2. 모바일/데스크톱에서 설치 가능
3. 설치 후 독립 창으로 실행됨
4. 로그인/권한/데이터 조회 기존 동작 유지
5. Vercel 배포 구조 유지
6. 캐시 때문에 이전 데이터나 이전 UI가 고정되는 문제 없음

## 12. 한 줄 요약

Phase 1은 "설치형 PWA"만 보수적으로 붙이는 단계다.  
앱처럼 설치는 가능하게 하되, 인증/API/동적 데이터는 캐시하지 않아 기존 백오피스 기능과 운영 흐름을 절대 깨지 않게 구현한다.
