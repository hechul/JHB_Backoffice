# 개발 계획서 — JHBioFarm 백오피스 v1

> 작성일: 2026-02-20
> 기준: PRD v5 + 스토리보드 + UI/UX 구현 완료 상태
> 목적: DB 설계 착수 전, 구현 범위와 순서를 확정한다.

---

## 1. 개발 방침

### 핵심 원칙: 뼈대를 먼저 세우고, 살을 붙인다

모든 기능을 한 번에 구현하지 않는다.
**핵심 파이프라인(업로드 → 필터링 → 고객 조회)을 최소 동작 수준으로 먼저 완성**하고,
이후 같은 화면에 기능을 추가해가며 점진적으로 고도화한다.

```
[뼈대]  업로드 → 필터링 → 고객 조회  (최소 동작)
  ↓
[살 붙이기 1차]  같은 화면에 고급 기능 추가 (수동 분류, 검증, 다운로드 등)
  ↓
[살 붙이기 2차]  대시보드, 실행 이력 등 나머지 화면 연동
  ↓
[살 붙이기 3차]  계정 관리, 권한 분기, 고도화 기능
```

새 화면을 추가하는 것이 아니라, **이미 만들어진 화면에 실 데이터와 기능을 채워넣는 방식**이다.

## 3. 우선순위 구조

### Phase 0 — 인프라 (전제 조건)

> 이게 없으면 아무것도 안 된다.

| 항목              | 이유                           |
| --------------- | ---------------------------- |
| **로그인/인증**      | RLS 정책상 인증 없이 DB 접근 불가       |
| **상품 마스터 (최소)** | 체험단 상품 매핑에 필수, 상품 없으면 필터링 불가 |

### Phase 1 — 핵심 파이프라인 (뼈대)

> 이 3개가 동작하면 서비스 가치가 발생한다.

| 순서  | 기능          | 한 줄 설명                    |
| --- | ----------- | ------------------------- |
| 1   | **데이터 업로드** | 주문 + 체험단 엑셀을 DB에 넣는다      |
| 2   | **체험단 분석**  | 5-Rank 필터링으로 체험단을 자동 판별한다 |
| 3   | **고객 조회**   | 실구매 고객을 필터링해서 엑셀로 뽑는다     |

### Phase 2 — 살 붙이기 (기존 화면 고도화 + 나머지 화면)

> Phase 1에서 만든 화면에 기능을 추가하고, 나머지 화면을 연동한다.

| 항목             | 설명                            |
| -------------- | ----------------------------- |
| 대시보드 실 데이터 연동  | KPI, 차트, 이탈 위험 고객             |
| 실행 이력 실 데이터 연동 | filter_logs, override_logs 조회 |
| 알림 센터          | 필터링 완료 알림                     |
| 양식 다운로드        | 주문/체험단 엑셀 템플릿                 |
| 상품 Soft Delete | 삭제 대신 비활성화                    |

### Phase 3 — 관리 + 고도화

> 운영 안정화 이후 추가한다.

| 항목                 | 설명                              |
| ------------------ | ------------------------------- |
| 계정 관리 (초대/비활성화)    | 현재는 Supabase 대시보드에서 직접 관리       |
| viewer/admin 권한 분기 | UI에서 viewer 변경 액션 차단            |
| 일괄 수동 분류           | 체크박스 다건 처리                      |
| 동시성 제어 (Lock)      | 필터링 동시 실행 방지                    |
| 성장 단계 분석           | max_stage 집계                    |
| 이탈 예측 고도화          | 평균 재구매 주기 기반 동적 기준              |
| 강제 매칭              | 미매칭 experience ↔ purchase 수동 연결 |
| 연관 구매 분석           | 함께 구매되는 상품 조합                   |

## 4. Phase 0 상세 — 인프라

### 4.1 로그인/인증

**구현:**

- Supabase Auth (Email + Password)
- 로그인 페이지 실 동작
- 세션 24시간 만료 + 자동 리디렉트
- RLS 정책 (authenticated만 접근)
- `profiles` 테이블

**제외 (나중에):**

- 회원가입 (초대 방식)
- 비밀번호 찾기
- viewer/admin 분기 (당분간 모두 admin)

**완료 기준:**

- [ ] 로그인 성공 → 홈 이동
- [ ] 인증 실패 → 에러 메시지
- [ ] 비로그인 → `/login` 리디렉트
- [ ] 로그아웃 동작

### 4.2 상품 마스터

**구현:**

- `products` 테이블 CRUD
- 목록 조회 + 검색
- 신규 등록 모달 + 수정 패널
- 실 상품 데이터 시딩 (최소 20건)

**제외 (나중에):**

- 일괄 업로드
- Soft Delete
- 성장 단계 필터

**완료 기준:**

- [ ] 상품 등록/수정/조회 정상 동작
- [ ] 실 상품 데이터 시딩 완료

## 5. Phase 1 상세 — 핵심 파이프라인

### 5.1 데이터 업로드

**구현:**

- 주문 파일 드래그앤드롭 업로드
- 체험단 파일 드래그앤드롭 업로드
- 헤더 기반 컬럼 매핑 (고정 인덱스 X)
- 필수 컬럼 검증 + 누락 컬럼명 표시
- 전처리: Trim, 날짜 통일, 옵션 정규화
- 주문상태 ETL 필터링 (유효 상태만 적재)
- UPSERT 처리 (purchases)
- 중복 방지 (experiences 복합 유니크)
- 업로드 결과 요약
- 캠페인 선택/생성
- 상품 매핑 (자동 + 수동 매핑 UI)
- 월 선택기 연동 (`all` 시 업로드 금지)

**나중에 살 붙일 것:**

- 양식 다운로드 버튼 → Phase 2
- 다중 파일 동시 업로드 → Phase 3
- 무효화 시 매칭 복구 → Phase 2

**필요한 DB:**

- `purchases`, `experiences`, `campaigns`

**완료 기준:**

- [ ] `.xlsx` 드래그앤드롭 업로드 동작
- [ ] 필수 컬럼 누락 시 차단 + 누락 목록 표시
- [ ] 취소/반품 주문 제외됨
- [ ] 동일 purchase_id 재업로드 시 UPDATE (중복 없음)
- [ ] 캠페인 미선택 시 업로드 차단
- [ ] 매핑 실패 건 → 매핑 UI 노출
- [ ] 결과 요약 (신규 N건 / 중복 N건 등) 표시

### 5.2 체험단 분석 (필터링)

**구현:**

- 필터링 시작 버튼 + 확인 모달
- 비동기 처리 (job_id → polling → 진행률)
- 5-Rank 순차 매칭 (PRD §3.3)
- 매칭 결과 저장 (is_fake, match_rank, match_reason, matched_exp_id)
- 미매칭 사유 자동 분류
- 수량 경고 (quantity ≥ 2)
- filter_ver 기록
- 결과 탭: 매칭 / 미매칭 / 확인 필요
- 수동 분류: fake↔real 전환 (건별)
- 수동 분류 시 experience 연결 자동 처리
- 수동 분류 이력 기록
- 재실행 시 is_manual 보존
- 월 선택기 연동 (`all` 시 실행 금지)
- 정리 CSV 다운로드

**나중에 살 붙일 것:**

- 일괄 수동 분류 (다건 처리) → Phase 3
- 동시성 제어 / Lock → Phase 3
- 강제 매칭 기능 → Phase 3
- 정리 파일 `.xlsx` 형식 → Phase 2

**필요한 DB:**

- `filter_jobs`, `filter_logs`, `override_logs`

**완료 기준:**

- [ ] 필터링 시작 → 진행률 → 완료 토스트
- [ ] Rank 1~5 매칭이 PRD 기준대로 동작
- [ ] 수동 분류 시 experience 연결 자동 해제/연결
- [ ] is_manual=true는 재실행 시 보존
- [ ] 결과 탭 전환 + 데이터 정상 표시
- [ ] CSV 다운로드 동작

### 5.3 고객 조회

**구현:**

- customers 집계 (필터링 완료 후 자동 갱신)
- 고객 목록 + 페이지네이션
- 필터: 검색, 펫 타입, 이탈 위험, 구매 횟수(1/3+/5+/10+)
- 필터 칩 + 전체 초기화
- 고객 상세 슬라이드 패널
- 엑셀 다운로드 (현재 필터 기준, 통합 1종)
- 딥링크 query 반영
- URL 쿼리 동기화

**나중에 살 붙일 것:**

- 성장 단계 필터 → Phase 3
- 마케팅용/분석용 엑셀 분리 → 불필요 (통합 유지)
- 이탈 예측 동적 기준 → Phase 3 (고정 60일만 적용)

**필요한 DB:**

- `customers`

**완료 기준:**

- [ ] 필터링 완료 후 customers 자동 집계
- [ ] 검색/필터 실시간 동작
- [ ] URL 쿼리 동기화
- [ ] 고객 상세 패널에 구매 이력 표시
- [ ] 엑셀 다운로드 동작
- [ ] 딥링크 (`/customers?churn=true`) 정상 동작

## 7. "살 붙이기" 전체 로드맵

각 화면이 Phase를 거치며 어떻게 성장하는지 한눈에 보는 표.

| 화면     | Phase 0 | Phase 1 (뼈대)    | Phase 2 (살 붙이기)  | Phase 3 (고도화)      |
| ------ | ------- | --------------- | ---------------- | ------------------ |
| 로그인    | 실 동작    | -               | -                | 비밀번호 찾기            |
| 홈      | -       | -               | 현황 요약 연동         | -                  |
| 업로드    | -       | 파일 업로드 + 매핑     | 양식 다운로드, 무효화 복구  | 다중 파일              |
| 체험단 분석 | -       | 필터링 + 수동 분류(건별) | xlsx 다운로드        | 일괄 분류, Lock, 강제 매칭 |
| 고객 조회  | -       | 필터 + 엑셀 + 딥링크   | -                | 성장 단계 필터, 이탈 동적 기준 |

| 계정 관리  | -       | -               | -                | 초대, 권한, 비활성화       |

## 8. Phase 1 완료 시 검증

Phase 1이 끝나면 아래를 실 데이터로 테스트한다:

```
1. 실제 스마트스토어 주문 엑셀 업로드 → DB 적재 확인
2. 실제 체험단 엑셀 업로드 → 캠페인 연결 + 상품 매핑 확인
3. 필터링 실행 → Rank별 매칭 결과 검증
4. 수동 분류 변경 → DB 반영 + 재실행 시 보존 확인
5. 고객 조회 → 필터 → 엑셀 다운로드 → 데이터 정합성 확인
6. 위 과정을 최소 2개월 데이터로 반복
```

이 테스트를 통과하면 Phase 2로 진입한다.
