# Attendance Push Reminder Plan

- 작성일: 2026-03-19
- 대상: `/attendance/records` 중심의 출퇴근 알림
- 목표: 휴대폰에서 퇴근 10분 전 `퇴근 10분 전입니다` 푸시 알림 받기

## 1. 결론

가능하다.

다만 이 기능은 브라우저 내부 타이머로 만드는 기능이 아니라 아래 4개가 함께 있어야 한다.

1. 설치형 PWA 또는 홈 화면 추가
2. 알림 권한 허용
3. 기기별 Web Push 구독 저장
4. 서버에서 정해진 시각에 푸시 발송

즉, "앱이 켜져 있을 때만 뜨는 알림"이 아니라 "설치된 웹앱에 서버가 보내는 푸시 알림" 구조로 가야 한다.

## 3. 가장 중요한 제약

### 3.1 iPhone

iPhone은 단순히 Safari 탭으로 접속한 상태만으로는 신뢰성 있게 이 기능을 쓸 수 없다.

필수 조건:

- 홈 화면에 추가된 웹앱이어야 함
- 사용자가 직접 알림 권한을 허용해야 함

이 부분은 WebKit이 iOS/iPadOS 16.4+ Home Screen web app에서 Web Push를 지원한다고 안내한다.

### 3.2 HTTPS

푸시 권한 요청과 Push 구독은 secure context에서만 가능하다.
즉 운영 배포 주소에서만 정상 동작한다고 봐야 한다.

### 3.3 Vercel Cron 제약

이 프로젝트는 현재 Vercel 배포다.

중요:

- Vercel Cron의 timezone은 UTC다.
- Hobby 플랜은 `하루 1회`만 가능하고, 시각 정밀도도 시간 단위라 10분 전 알림에 부적합하다.
- Pro 이상이어야 분 단위 스케줄이 현실적이다.

따라서 운영 Vercel 플랜이 Hobby면, 분 단위 외부 스케줄러를 따로 써야 한다.

## 4. 추천 구현 방식

### 4.1 전체 구조

1. 사용자가 `/attendance/records`에서 `퇴근 알림 받기` 버튼 클릭
2. 브라우저가 알림 권한 요청
3. 서비스 워커의 `PushManager.subscribe()`로 기기 구독 생성
4. 구독 정보를 DB에 저장
5. 서버 cron이 매분 실행
6. 서버가 "오늘 아직 퇴근 안 했고, 퇴근 예정 10분 전인 사람"을 조회
7. 대상자에게 Web Push 발송
8. 서비스 워커가 시스템 알림 표시
9. 알림 클릭 시 `/attendance/records`로 이동
10. 동시에 앱 내부 `notifications` 테이블에도 같은 메시지를 남김

### 4.2 왜 이 방식이 좋은가

- 기존 PWA 구조를 그대로 활용 가능
- 현재 `notifications` UI와도 자연스럽게 연결 가능
- 로그인/권한/근태 로직을 크게 건드리지 않음
- 문제 생기면 push 관련 범위만 롤백 가능

## 5. 데이터 모델 제안

### 5.1 `web_push_subscriptions`

용도:

- 기기별 푸시 구독 저장
- 한 사용자가 휴대폰/태블릿/맥북 여러 대를 쓸 수 있게 함

권장 컬럼:

- `id uuid primary key default gen_random_uuid()`
- `user_id uuid not null references public.profiles(id) on delete cascade`
- `endpoint text not null`
- `p256dh text not null`
- `auth text not null`
- `device_label text`
- `user_agent text`
- `permission_state text not null default 'granted'`
- `is_active boolean not null default true`
- `last_seen_at timestamptz not null default now()`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

권장 제약:

- `unique(endpoint)`
- `permission_state in ('granted', 'denied', 'default')`

### 5.2 `attendance_push_preferences`

용도:

- 사용자가 알림을 켰는지/껐는지 저장
- 나중에 `10분 전`, `5분 전`, `출근 시간` 같은 확장 대비

권장 컬럼:

- `user_id uuid primary key references public.profiles(id) on delete cascade`
- `enabled boolean not null default false`
- `minutes_before_checkout integer not null default 10`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

v1에서는 `minutes_before_checkout`를 10으로 고정해도 된다.

### 5.3 `attendance_push_delivery_logs`

용도:

- 같은 날 같은 사용자에게 중복 발송 방지
- 실패 구독 정리 근거 저장

권장 컬럼:

- `id bigint generated by default as identity primary key`
- `user_id uuid not null references public.profiles(id) on delete cascade`
- `subscription_id uuid references public.web_push_subscriptions(id) on delete set null`
- `work_date date not null`
- `reminder_type text not null`
- `scheduled_for timestamptz not null`
- `sent_at timestamptz`
- `status text not null default 'pending'`
- `error_message text`
- `created_at timestamptz not null default now()`

권장 제약:

- `unique(user_id, work_date, reminder_type)`
- `reminder_type in ('checkout_minus_10m')`
- `status in ('pending', 'sent', 'failed', 'skipped')`

## 6. 대상자 판별 규칙

v1은 보수적으로 간다.

### 6.1 발송 대상

- 오늘 출근 기록이 있음
- 오늘 퇴근 기록이 아직 없음
- 푸시 알림 설정이 켜져 있음
- 활성 구독이 1개 이상 있음
- 오늘이 주말이 아님
- 승인된 종일 부재가 아님
- 아직 같은 알림을 오늘 발송하지 않음

### 6.2 v1에서 제외

- 반차별 세밀한 퇴근 시각 계산
- 개인별 다른 근무시간
- 교대근무
- 점심시간/일시중단을 반영한 동적 "예상 퇴근 시각"

이 프로젝트는 현재 `attendance_settings`가 전사 공통 1세트 구조다.
따라서 v1의 기준 시각은 `attendance_settings.work_end_time - 10분`이 가장 안전하다.

### 6.3 반차 처리

반차는 v1에서 제외하는 것이 안전하다.

이유:

- `half_am`, `half_pm`의 실제 퇴근 기준 시각이 현재 스키마에 명시되어 있지 않음
- 잘못 보내면 사용자가 가장 불편함

즉 v1 규칙:

- `annual`, `sick`, `official`, `other`, `half_am`, `half_pm` 모두 승인 상태면 알림 제외

## 7. 서버 구조 제안

### 7.1 API 엔드포인트

#### `POST /api/push/subscribe`

역할:

- 로그인 사용자의 PushSubscription 저장/갱신

입력:

- `subscription`
- `deviceLabel`

동작:

- 세션 사용자 확인
- endpoint/p256dh/auth 추출
- `web_push_subscriptions` upsert
- `attendance_push_preferences.enabled = true` 가능

#### `POST /api/push/unsubscribe`

역할:

- 해당 endpoint 비활성화

#### `POST /api/push/test`

역할:

- 수동 테스트 발송
- 개발/운영 검증용

#### `GET /api/cron/attendance-reminders`

역할:

- 매분 실행되는 발송 엔드포인트

보안:

- `CRON_SECRET` 검사
- 또는 `vercel-cron/1.0` user-agent 검사
- 둘 다 쓰는 것이 안전

### 7.2 서버 발송 로직

1. 현재 시각을 KST 기준으로 계산
2. 오늘 날짜 계산
3. `attendance_settings`에서 기준 퇴근 시각 조회
4. 오늘 알림 대상자 조회
5. 오늘 동일 알림 이미 보냈는지 확인
6. 각 구독으로 푸시 발송
7. 성공 시 `attendance_push_delivery_logs` 기록
8. 동시에 `notifications` 테이블에도 in-app 알림 insert
9. 404/410 구독은 `is_active=false` 처리

## 15. 추천 결론

이 프로젝트에서는 아래 순서가 가장 현실적이다.

1. `퇴근 10분 전 푸시`만 v1 범위로 고정
2. `반차/개인 근무시간`은 v1에서 제외
3. 먼저 `수동 테스트 푸시`까지 구현
4. 기기 검증 후 `자동 cron 발송` 활성화

추가 판단:

- Vercel이 Hobby면 minute-level cron 때문에 바로 막힌다.
- Pro 이상이면 현재 구조에서 그대로 구현 가능하다.
