  - 계정관리(`/settings/users`): 목업 데이터 기반, 사이드바 `준비중`

  - 핵심 파이프라인(업로드/필터링/고객분석/실행이력): 실데이터 동작 완료

### 목표

3. 고급 예측 모델(동적 이탈 예측, 연관구매 추천 자동화)

## 2) 트랙 A — 계정관리 개발 플랜

## A-1. 정책 확정 (선행)

### 확정 정책
1. `admin`: 계정 초대/권한 변경/비활성화 가능
2. `modifier`: 데이터 작업 권한은 가능, 계정 초대/권한 변경은 불가
3. `viewer`: 조회 전용
4. 초기 운영 계정은 2개(대표 `modifier`, 관리자 `admin`)를 기본 유지

### 정책 체크 포인트
1. 본인 계정 강등/비활성화 방지
2. 마지막 `admin` 계정 비활성화 방지
3. 공용 대표 계정 사용 시 변경 기록에 `변경자명`을 남기는 UX 유지

## A-2. 백엔드/보안

### 구현 항목
1. `profiles` 조회/수정 RLS 점검 (`admin`만 role/status 변경 가능)
2. 계정관리 전용 RPC 또는 서버 라우트 도입
   - `set_user_role(user_id, role)`
   - `set_user_status(user_id, status)`
3. 변경 이력 저장
   - 권장 테이블: `account_audit_logs` (who/when/target/before/after)
4. 초대 기능은 2단계로 분리
   - 1차: UI 비활성(문구로 “후속”)
   - 2차: 서버 경유 초대(`auth.admin.inviteUserByEmail`)

### DB 변경(예정)
1. `account_audit_logs` 신규 테이블
2. 필요 시 `profiles`에 표시용 컬럼 보강(예: `updated_at`)

## A-3. 프론트엔드

### 구현 항목
1. `/settings/users` 목업 배열 제거 → `profiles` 실조회
2. 역할 배지 3종(`admin`/`modifier`/`viewer`) 표시
3. 상태 토글(활성/비활성) 실반영
4. 역할 변경 액션(드롭다운 또는 모달) 추가
5. 실패 케이스 UX
   - 권한 없음
   - 마지막 admin 보호 정책 위반
   - 네트워크/DB 오류
6. 사이드바 `계정 관리`의 `준비중` 제거 시점
   - A-2/A-3 완료 후 오픈

## A-4. 완료 기준 (DoD)
1. 실데이터 사용자 목록 조회 가능
2. `admin`만 역할/상태 변경 가능
3. 변경 시 이력 저장 확인 가능
4. `modifier`/`viewer`에서 제한 동작 정상
5. 새로고침 후 권한/목록 상태 일관성 유지

## 3) 트랙 B — 대시보드 개발 플랜

## B-1. 지표 정의 고정

### KPI 정의
1. 전체 주문: `purchases` 월 기준 건수
2. 실구매: `is_fake = false` 건수
3. 재구매 고객: 월 내 고객별 구매횟수 2회 이상 고객 수
4. 신규 고객: 기준월 첫 구매 고객 수

### 차트/리스트 정의
1. 월별 실구매 추이: 최근 N개월 실구매 건수
2. 펫 타입 분포: 실구매 기준 DOG/CAT/BOTH 비율
3. 인기 상품 TOP 5: 실구매 기준 상품 건수 상위
4. 이탈 위험 고객: 최근 주문일 90일 초과 고객

## B-2. 데이터 계층

### 구현 방식
1. 1차: RPC 2~4개로 서버 집계
   - `get_dashboard_kpi(month)`
   - `get_dashboard_trend(month_range)`
   - `get_dashboard_top_products(month)`
   - `get_dashboard_churn(month)`
2. 2차(선택): 단일 RPC로 통합해 요청 수 축소

### 성능 기준
1. 대시보드 첫 진입 2초 내 주요 KPI 표시
2. 월 전환 시 1.5초 내 KPI 갱신
3. 쿼리 오류 시 섹션 단위 graceful fallback

## B-3. 프론트엔드

### 구현 항목
1. `/dashboard` 목업 데이터 제거, 실데이터 바인딩
2. 월 선택기 연동 (`selectedMonth`)
3. KPI 카드 딥링크 유지 (`/customers` 쿼리 전달)
4. 로딩/에러/빈 상태 UI 분리
5. 사이드바 `대시보드`의 `준비중` 제거 시점
   - B-2/B-3 완료 후 오픈

## B-4. 완료 기준 (DoD)
1. KPI/차트/리스트가 DB 기준으로 갱신
2. 월 변경 시 전체 컴포넌트 일관 갱신
3. 카드 클릭 딥링크 정확 동작
4. 새로고침/재진입 시 목업 데이터 노출 없음
