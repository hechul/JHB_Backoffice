# Backoffice v2 GoodForPet 고도화 방안

- 작성일: 2026-04-06
- 기준 브랜치: `dev/naver-commerce-api`
- 기준 앱: `apps/backoffice-v2`
- 관련 축:
  - 네이버 커머스 API 기반 주문 수집
  - 쿠팡 주문/정산 연동
  - 향후 네이버 검색광고 API 기반 광고 성과 확장

---

## 1. 문서 목적

이 문서는 `apps/backoffice-v2`를 굿포펫 운영 백오피스에서 한 단계 더 끌어올리기 위한 기준 계획서다.

목표는 아래 3가지를 동시에 만족하는 것이다.

1. 운영자가 매일 보는 화면이 더 빠르게 읽혀야 한다.
2. 금액/정산/재구매 지표가 신뢰 가능한 상태로 보여야 한다.
3. 향후 `네이버 검색광고 API`를 붙였을 때도 구조가 흔들리지 않아야 한다.

이 문서는 새 차트를 많이 붙이는 계획서가 아니다.
우선순위는 `데이터 신뢰도 -> 경영 요약 고도화 -> 채널/상품/고객/리텐션 고도화 -> 광고 데이터 확장` 순서다.

---

## 2. 현재 상태 진단

### 2-1. 이미 잘 되어 있는 부분

- 분석 화면 축이 이미 `경영 요약 / 채널 성과 / 상품 성과 / 고객 구성 / 재구매 흐름`으로 나뉘어 있다.
- `purchases`에 `payment_amount`, `expected_settlement_amount`, `payment_commission`, `sale_commission`이 들어갈 수 있는 구조가 준비되어 있다.
- 네이버/쿠팡을 `source_channel`, `source_fulfillment_type`으로 분리해 확장할 수 있다.
- 상품/고객/재구매 쪽의 기본 로직은 이미 굿포펫 운영 맥락에 맞게 커스터마이즈되어 있다.

### 2-2. 현재 가장 큰 문제

- `주문수`와 `결제금액`의 반영률이 서로 다른데도 화면에서 그 차이를 크게 드러내지 못한다.
- 쿠팡은 `marketplace`와 `rocket_growth`가 날짜 기준과 금액 채움 상태가 다를 수 있다.
- 네이버는 비브랜드스토어 제약 때문에 공식 통계센터 수준의 유입/행동/고객 통계 API를 그대로 붙일 수 없다.
- 현재 구조는 “분석 화면”은 있지만 “숫자를 믿어도 되는가”를 알려주는 계층이 약하다.
- 향후 검색광고 데이터를 붙일 때, 커머스 결제 데이터와 광고 전환 데이터를 같은 숫자처럼 오해할 위험이 있다.

## 3. 작명과 IA 원칙

### 3-1. 문구 원칙

- `대표 보고`, `대표용`, `대표님용` 같은 표현은 UI에서 제거한다.
- 화면 이름은 역할이 바로 읽히는 명사형으로 유지한다.
- 권장 그룹명:
  - `분석`
  - `데이터 운영`
  - `기준 정보`

### 3-2. 권장 IA

- `경영 요약`
- `채널 손익`
- `상품 인텔리전스`
- `고객 세그먼트`
- `재구매 리텐션`

- `주문 동기화`
- `데이터 신뢰도`
- `필터링`
- `실행 이력`

- `상품 목록`

### 3-3. 왜 이렇게 나누는가

- `경영 요약`은 판단용이다.
- `채널/상품/고객/리텐션`은 원인 분석용이다.
- `주문 동기화/데이터 신뢰도`는 숫자를 믿을 수 있게 만드는 운영용이다.
- 따라서 “숫자를 읽는 화면”과 “숫자를 고치는 화면”은 절대 섞지 않는다.

## 5. 데이터 신뢰도 레이어

### 5-1. 왜 최우선인가

금액형 화면은 예쁘게 만드는 것보다 먼저 `신뢰도`가 확보되어야 한다.

예:

- 주문수는 177건인데 결제금액은 71건만 반영된 상태
- 네이버 raw 데이터에는 금액이 있는데 `purchases` projection만 비어 있는 상태
- 쿠팡 `rocket_growth`는 채워졌는데 `marketplace`나 네이버가 덜 채워진 상태

이 상황에서 경영 화면이 숫자를 확정값처럼 보이면 잘못된 판단을 유도한다.

### 5-2. 필수 규칙

- `금액 반영률 < 80%`
  - 금액 KPI는 `집계 보류`
  - 빨간 배지 표시
- `80% <= 반영률 < 98%`
  - 숫자는 보여주되 `부분 반영`
  - 노란 배지 표시
- `반영률 >= 98%`
  - `신뢰 가능`
  - 초록 배지 표시

### 5-3. 꼭 보여줘야 하는 정보

- `마지막 성공 동기화 시각`
- `결제 반영률`
- `정산 반영률`
- `최근 실패 여부`
- `실패 원인`
- `영향 받은 월`
- `백필 필요 월`

### 5-4. 새 페이지

`데이터 신뢰도` 페이지를 추가한다.

상단 위젯:

- `금액 누락 월`
- `채널별 반영률`
- `미매핑 상품`
- `최근 7일 실패 동기화`
- `확인 필요 범위`

## 7. 네이버 검색광고 API 확장 계획

### 7-1. 왜 붙이는가

현재 커머스 데이터만으로는 `결과`는 볼 수 있어도 `왜 그렇게 되었는지`를 충분히 설명하기 어렵다.

검색광고 API를 붙이면 아래 질문에 답할 수 있다.

- 어떤 캠페인이 검색량/클릭을 만들었는가
- 어떤 키워드가 효율이 좋은가
- 어떤 상품군이 광고 지출 대비 성과가 나오는가

### 7-2. 적용 원칙

- 검색광고 데이터는 `커머스 결제 데이터`와 분리 저장한다.
- 광고의 `전환`, `매출`, `ROAS` 같은 수치는 attribution 기준이 다르므로, 결제금액과 직접 합산하지 않는다.
- 초기에는 `읽기 전용 리포트`부터 시작한다.
- 광고 데이터가 안정화되기 전에는 사이드바에 정식 메뉴로 노출하지 않는다.

### 7-3. 우선 사용할 범위

1. `계정/캠페인/광고그룹/키워드` 마스터 동기화
2. `일별 성과 리포트`
3. `키워드 도구(연관 키워드/검색량)` 기반 인사이트

### 7-4. 제안 테이블

- `naver_searchad_accounts`
- `naver_searchad_sync_runs`
- `naver_searchad_campaigns`
- `naver_searchad_adgroups`
- `naver_searchad_keywords`
- `naver_searchad_daily_stats`
- `naver_searchad_keyword_stats`
- `naver_searchad_related_keywords_cache`

### 7-5. 제안 화면

#### 광고 성과

- `광고비`
- `노출수`
- `클릭수`
- `CTR`
- `평균 클릭비용`
- `전환수`
- `광고 기준 매출`
- `ROAS`

#### 검색어 인사이트

- `상위 유입 키워드`
- `고효율 키워드`
- `비효율 키워드`
- `상품군별 키워드 매핑`
- `연관 키워드 제안`

### 7-6. 절대 섞지 말아야 할 것

- 검색광고 `전환매출`과 커머스 `결제금액`
- 검색광고 `ROAS`와 실제 정산 기준 순매출
- 키워드 기준 광고 성과와 상품 기준 결제금액을 같은 차트에서 기본값으로 합산

광고 화면은 별도 기준 배지를 강제한다.

- `광고기준`
- `커머스기준`

## 8. 구현 단계

### Phase 0 — 지금 바로 반영

- `대표 보고` 같은 문구 제거
- `데이터 신뢰도` 페이지/공통 스트립 추가
- 금액 반영률 계산 도입
- 분석 페이지 상단에 데이터 상태 노출

### Phase 1 — 금액형 분석 안전화

- `경영 요약`, `채널 성과`, `상품 성과`를 coverage-aware로 개편
- 반영률 부족 시 `집계 보류` 규칙 적용
- 금액 기반 전월 비교 차단 규칙 추가

### Phase 2 — 채널/상품/고객 고도화

- `채널 손익` 구조 반영
- 상품 인텔리전스 세분화
- 고객 세그먼트 고도화
- 리텐션 cohort 고도화

### Phase 3 — 검색광고 읽기 전용

- 검색광고 인증/동기화 구조 추가
- 캠페인/광고그룹/키워드 마스터 수집
- 일별 광고 성과 적재

### Phase 4 — 광고와 커머스의 병렬 분석

- `광고 성과` 페이지 추가
- `검색어 인사이트` 페이지 추가
- 상품/채널 페이지에서 광고 영향 보조 패널 제공

### Phase 5 — 신뢰 가능한 스냅샷

- 마감 월은 `trusted snapshot`으로 고정
- 진행 중인 월만 live 모드 제공
- 경영 화면 기본값은 snapshot 우선
