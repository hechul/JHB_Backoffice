## 1. 이 문서의 목적

이 문서는 네이버 커머스 API 주문을 받아왔을 때, `외부 상품번호`를 기준으로 우리 백오피스의 내부 상품에 어떻게 연결할지를 설명하는 문서다.

중요한 점은 이 문서가 `구현 용어`를 설명하는 문서가 아니라는 것이다.

이 문서에서 정리하려는 것은 아래 4개다.

- 어떤 기준으로 상품을 먼저 찾을 것인지
- 어떤 상품은 상품번호만으로 충분한지
- 어떤 상품은 옵션까지 같이 봐야 하는지
- 어떤 상품은 상품명까지 다시 읽어야 하는지

즉 이 문서의 핵심은 아래 한 줄이다.

- `상품번호로 먼저 찾고, 필요할 때만 옵션과 상품명을 보정에 사용한다.`

## 2. 왜 상품번호 중심으로 가야 하는가

지금까지는 엑셀 업로드 기준이라 상품명과 옵션명을 정규화해서 내부 상품에 연결하는 방식이 중심이었다.

이 방식이 필요했던 이유는 분명했다.

- 체험단 데이터에는 네이버 상품번호가 없다.
- 엑셀마다 상품명이 조금씩 다르다.
- 그래서 내부 상품목록의 이름을 기준으로 정규화가 필요했다.

하지만 네이버 커머스 API를 붙이면 상황이 달라진다.

- 주문행마다 `상품번호`가 들어온다.
- 실제 2025년 12월, 2026년 1월, 2026년 2월 주문 데이터를 보면, 상품명은 자주 바뀌어도 상품번호는 비교적 안정적이다.
- 따라서 API 주문을 내부 상품에 연결할 때는 `상품명`보다 `상품번호`를 먼저 믿는 것이 훨씬 안전하다.

정리하면:

- 상품명은 흔들린다.
- 상품번호는 상대적으로 덜 흔들린다.
- 따라서 API 주문 매핑의 시작점은 상품번호가 맞다.

---

## 3. 그래도 내부 상품명은 계속 중요하다

상품번호를 먼저 본다고 해서, 백오피스 안에서 상품명을 상품번호로만 관리하겠다는 뜻은 아니다.

백오피스 안에서는 계속 아래 기준이 유지되어야 한다.

- 내부 상품목록의 상품명은 우리가 분석에 쓰는 기준 이름이다.
- 대시보드, 고객분석, 체험단 필터링, 성장단계는 내부 상품명을 기준으로 돌아간다.
- 따라서 최종 저장 값은 계속 내부 상품목록의 이름으로 정규화되어야 한다.

즉 역할을 나누면 이렇다.

- 외부 상품번호: 어떤 상품군인지 찾는 첫 단서
- 내부 상품명: 최종적으로 시스템 안에 저장할 canonical 이름

이 둘은 같은 역할이 아니다.

## 4. 전체 매핑 흐름

네이버 API로 주문 1건이 들어왔을 때, 매핑은 아래 순서로 판단한다.

1. 주문행의 상품번호를 본다.
2. 그 상품번호가 우리 내부에서 어떤 상품군인지 찾는다.
3. 해당 상품이 옵션 구분형이면 옵션을 같이 본다.
4. 해당 상품이 옵션만으로는 안 되면 상품명도 같이 본다.
5. 그래도 확정이 안 되면 기존 이름 정규화 로직으로 한 번 더 본다.
6. 마지막까지 확정이 안 되면 자동 확정하지 않고 검수 대상으로 남긴다.

즉 항상 아래 순서다.

- 1차: 상품번호
- 2차: 옵션
- 3차: 상품명
- 4차: 기존 정규화 fallback
- 5차: 수동 검수

## 5. 상품을 크게 4종류로 나눈다

네이버 상품은 실제로 모두 같은 방식으로 매핑되지 않는다.

운영상 아래 4종류로 나눠서 보는 것이 가장 안전하다.

### 5-1. 상품번호만 보면 되는 상품

이 그룹은 상품번호만으로도 내부 상품이 안정적으로 확정된다.

이런 상품은:

- 상품명이 마케팅 문구에 따라 바뀌더라도
- 상품번호가 같으면 같은 상품으로 본다.

대표 예:

- 두부모래
- 케어푸
- 엔자이츄 묶음별 SKU
- 이즈바이트 묶음별 SKU
- 리뉴얼 전 동결건조
- 애착트릿 3종세트
- 일부 프로모션 번들 SKU

### 5-2. 상품번호와 옵션을 같이 봐야 하는 상품

이 그룹은 상품번호 하나 아래에 여러 종류가 같이 들어 있다.

즉 상품번호만으로는 “어느 variant인지” 확정할 수 없다.

대표 예:

- 츄라잇
- 전제품 맛보기 샘플
- 도시락 샘플팩
- 츄르짜개
- 미니 트릿백

이 상품들은 상품번호로 먼저 상품군을 찾고, 옵션으로 세부 구분을 한다.

### 5-3. 상품번호로는 상품군만 찾고, 최종 판단은 상품명까지 읽어야 하는 상품

이 그룹은 상품번호도 도움이 되지만, 상품번호만으로 variant를 확정하면 위험하다.

대표 예:

- 애착트릿 리뉴얼 SKU

이 경우는 상품번호로 먼저 “애착트릿 리뉴얼군”을 잡고, 그 다음 상품명과 옵션에서 `북어`, `연어`, `치킨` 같은 키워드를 읽어 최종 맛을 결정해야 한다.

### 5-4. 자동 판단이 위험한 상품

아래는 자동 확정을 바로 하면 안 된다.

- 새로 들어온 프로모션 SKU
- 처음 보는 상품번호
- 하나의 상품번호 안에 예외가 너무 많은 경우
- 옵션이 비어 있거나 여러 의미로 읽히는 경우

이 경우는 검수 대상으로 남기는 것이 맞다.

## 6. 상품번호만으로 매핑 가능한 상품

아래는 2025년 12월부터 2026년 2월까지 실제 주문 데이터 기준으로, 상품번호만으로도 안정적으로 볼 수 있는 상품들이다.

### 6-1. 두부모래

- `11749388704` = 1개
- `11750103214` = 3개
- `11750107226` = 6개

판단 방식:

- 상품명이 `고양이두부모래`, `프리미엄 두부모래`, `먼지없는 가는입자`처럼 바뀌어도
- 상품번호가 같으면 같은 내부 상품으로 본다.

주의:

- 두부모래는 옵션을 매핑 기준으로 사용하지 않는다.
- 묶음 수는 상품번호가 이미 설명하고 있다.

### 6-2. 케어푸

- `12320752331` = 기본 30포 상품
- `13041293177` = 90포 3개 상품

판단 방식:

- `포스트바이오틱스`, `유산균`, `오메가3`, `설날` 같은 문구가 붙어도
- 상품번호가 같으면 같은 상품군으로 본다.

### 6-3. 엔자이츄

- `12565223228` = 1개 SKU
- `13031643891` = 2개 SKU
- `13031643892` = 3개 SKU

판단 방식:

- 엔자이츄는 상품번호가 곧 묶음 단위를 구분해준다.
- 따라서 상품번호만으로 어느 묶음인지 확정할 수 있다.

중요한 주의사항:

- `1개짜리 상품 2개 구매`와 `2개짜리 상품 1개 구매`는 다르다.
- 두 경우는 상품번호가 다르다.
- 따라서 단순히 수량 숫자만 보고 같은 주문으로 취급하면 안 된다.

### 6-4. 이즈바이트

- `12565154404` = 1개 SKU
- `13035043593` = 2개 SKU
- `13035043594` = 3개 SKU

판단 방식:

- 엔자이츄와 동일하다.
- 상품번호로 묶음 단위를 구분한다.

### 6-5. 리뉴얼 전 동결건조

- `11034170709`
- `11034374158`
- `11034381845`
- `11687327189`

판단 방식:

- 리뉴얼 후 애착트릿과 섞지 않는다.
- 리뉴얼 전 동결건조는 별도 상품군으로 유지한다.

### 6-6. 애착트릿 3종세트

- `12825618625`

판단 방식:

- 단품 맛 구분 로직에 섞지 않는다.
- 3종세트는 별도 상품으로 관리한다.

### 6-7. 프로모션 번들 SKU

- `13074780587` = 츄라잇 3박스 프로모션 SKU

판단 방식:

- 일반 츄라잇 기본 상품과 분리한다.
- 프로모션 전용 SKU는 프로모션 SKU로 따로 본다.

## 7. 상품번호와 옵션을 같이 봐야 하는 상품

### 7-1. 츄라잇

- `12417368947`

판단 방식:

- 상품번호로 먼저 `츄라잇` 상품군을 찾는다.
- 그 다음 옵션으로 어떤 세부 variant인지 구분한다.

왜 옵션이 필요한가:

- 츄라잇은 내부적으로 `데일리핏`, `브라이트`, `클린펫` 같은 옵션 구분이 중요하다.
- 현재 로직도 츄라잇은 옵션 원문을 축약하지 않고 그대로 유지한다.

즉 츄라잇은 아래처럼 간다.

- 상품번호로 `층`을 잡고
- 옵션으로 최종 variant를 정한다.

추가 원칙:

- 츄라잇은 옵션값이 비어 있으면 자동 확정하지 않는다.
- 상품명만 보고 `데일리핏`, `브라이트`, `클린펫`을 임의로 추정하지 않는다.
- 옵션이 비어 있는 주문행은 기존 이름 정규화의 참고 후보로만 두고, 단일 근거가 없으면 검수 대상으로 남긴다.

### 7-2. 전제품 맛보기 샘플

- `12668256525`

판단 방식:

- 상품번호로 먼저 `전제품 맛보기 샘플` 상품군을 찾는다.
- 그 다음 옵션으로 실제 샘플 구성을 읽는다.

실제 확인된 예:

- 애착트릿 3종
- 츄라잇 3종
- 케어푸 강아지 유산균 3개
- 이즈바이트 1개
- 엔자이츄 1개

중요한 점:

- 이 상품은 본상품을 뜻하지 않는다.
- 예를 들어 `애착트릿 샘플`이 들어왔다고 해서 `애착트릿 본상품 주문`으로 바꾸면 안 된다.
- 샘플은 샘플로 남겨야 한다.

### 7-3. 도시락 샘플팩

- `12668454235`

판단 방식:

- 상품번호로 먼저 도시락 샘플 상품군을 찾는다.
- 그 다음 옵션으로 `강아지용`, `고양이용`을 구분한다.

주의:

- 현재 데이터에서는 `고양이 도시락`만 명확히 확인됐더라도,
- 상품명 자체가 `(강아지/고양이)`이므로 옵션형 상품으로 설계하는 것이 안전하다.

### 7-4. 츄르짜개

- `12668877332`

판단 방식:

- 상품번호로 먼저 츄르짜개 상품군을 찾는다.
- 그 다음 옵션으로 색상을 구분한다.

확인된 옵션:

- 옐로
- 블루
- 퍼플

주의:

- 현재 체험단 필터링 로직에서는 츄르짜개의 옵션 비교를 생략한다.
- 하지만 주문 정규화 단계에서는 옵션을 저장해두는 것이 맞다.

### 7-5. 미니 트릿백

- `12673164727`

판단 방식:

- 상품번호로 먼저 미니 트릿백 상품군을 찾는다.
- 그 다음 옵션으로 색상을 구분한다.

확인된 옵션:

- 민트
- 퍼플

## 8. 상품명까지 같이 읽어야 하는 예외 상품

### 애착트릿 리뉴얼

관련 상품번호:

- `12825519864`
- `12825541776`
- `12825547641`

이 상품군은 주의가 필요하다.

겉으로 보면:

- 하나는 북어
- 하나는 연어
- 하나는 치킨

처럼 보이지만, 실제 데이터에는 맛 표기가 서로 섞인 행이 있다.

실제 예:

- `12825519864` 안에 `애착트릿 연어 110g`
- `12825541776` 안에 `애착트릿 북어 100g`
- `12825547641` 안에 `애착트릿 북어 100g`
- `12825547641` 안에 `애착트릿 연어 110g`

따라서 애착트릿은 아래처럼 판단해야 한다.

1. 상품번호로 먼저 `애착트릿 리뉴얼 상품군`이라는 사실만 잡는다.
2. 옵션정보에 `북어`, `연어`, `치킨`이 있으면 옵션을 우선 본다.
3. 옵션에 없으면 상품명에서 `북어`, `연어`, `치킨`, `닭가슴살`, `닭고기`, `황태`를 읽는다.
4. 그래도 확정이 안 되면 자동 확정하지 않는다.

즉 애착트릿의 핵심은 이렇다.

- 상품번호는 첫 단서일 뿐이다.
- 최종 맛 판단은 상품명/옵션으로 다시 해야 한다.

## 9. 수량은 어떻게 볼 것인가

이 부분은 매우 중요하다.

네이버 API에서 들어오는 수량은 “그 SKU가 몇 개 주문됐는가”에 가깝다.

예를 들어:

- `1개짜리 엔자이츄 SKU`를 2개 사면 API 수량은 `2`
- `2개짜리 엔자이츄 SKU`를 1개 사면 API 수량은 `1`

즉 API 수량만 보고 실제 개수를 판단하면 안 된다.

### 9-1. 왜 상품명 파싱만으로는 위험한가

현재 엑셀 기반 로직은 상품명에서 `10개`, `20개`, `30개` 같은 숫자를 읽는 휴리스틱을 쓰고 있다.

하지만 실제 데이터에서는 같은 상품번호 안에서 아래 같은 흔들림이 있다.

- 엔자이츄 1개 SKU인데 상품명에는 `1개` 또는 `10개`처럼 보이는 표현이 섞임
- 엔자이츄 2개 SKU인데 상품명에는 `2개` 또는 `20개`처럼 보이는 표현이 섞임
- 이즈바이트도 비슷하게 `1개/7개`, `2개`, `3개` 표현이 섞임

즉 raw 상품명만 믿고 수량을 읽으면 10배, 7배로 잘못 계산될 위험이 있다.

### 9-2. v2에서의 수량 원칙

v2에서는 아래처럼 본다.

- API 수량은 원본 수량으로 그대로 저장
- 실제 분석용 수량은 상품번호에 연결된 묶음 정보로 다시 계산
- 즉 `원본 수량`과 `분석용 계산 수량`은 같은 값으로 덮어쓰지 않고, 서로 다른 의미의 값으로 관리한다.

예:

- 엔자이츄 1개 SKU -> 묶음값 1
- 엔자이츄 2개 SKU -> 묶음값 2
- 엔자이츄 3개 SKU -> 묶음값 3

최종 계산 개념:

- `실제수량 = API수량 x 상품번호별 묶음값`

같은 방식으로:

- 이즈바이트
- 두부모래
- 프로모션 번들

도 묶음 단위를 상품번호 기준으로 해석할 수 있다.

### 9-3. 정리

수량은 아래 순서로 보는 것이 맞다.

1. API 수량을 원본 그대로 저장
2. 상품번호가 어떤 묶음 SKU인지 확인
3. 그 묶음값으로 실제 분석 수량을 계산

즉 v2에서는 `상품명 파싱`보다 `상품번호별 묶음 정의`가 더 중요하다.

## 10. 기존 이름 정규화 로직은 어떻게 활용할 것인가

기존 로직은 버리지 않는다.

다만 역할이 바뀐다.

기존에는:

- 이름 정규화가 1차 기준이었다.

v2에서는:

- 이름 정규화는 fallback 기준이 된다.

활용 방식은 아래와 같다.

- 상품번호로 확정되면 그대로 내부 상품에 연결
- 상품번호 기준이 비어 있거나 애매하면 기존 정규화 로직으로 한 번 더 본다
- 이 fallback으로 단일 후보가 나오면 자동 연결을 허용할 수 있다
- 그래도 안 되면 검수 대상으로 남긴다

즉 기존 정규화 로직은 사라지는 것이 아니라, `주력`에서 `보조 안전장치`로 역할이 바뀐다.

## 11. 자동 확정하지 말아야 하는 경우

아래는 자동으로 상품을 정하지 않는 것이 맞다.

- 처음 보는 상품번호가 들어왔을 때
- 하나의 상품번호에 서로 다른 의미가 섞여 있을 때
- 옵션이 비어 있어서 어떤 variant인지 알 수 없을 때
- 애착트릿처럼 상품명/옵션을 봐도 맛이 확정되지 않을 때
- 프로모션 상품이 일반 상품으로 잘못 흡수될 위험이 있을 때

이 경우는 아래처럼 처리한다.

- 원본 주문행은 저장
- 내부 상품 연결은 보류
- 운영자가 검수 후 연결

## 12. 초기 입력 우선순위

실제 적용은 아래 순서로 가는 것이 가장 안전하다.

1. 두부모래
2. 케어푸
3. 엔자이츄
4. 이즈바이트
5. 츄라잇
6. 애착트릿
7. 전제품 맛보기 샘플
8. 도시락 샘플팩
9. 츄르짜개
10. 미니 트릿백
11. 리뉴얼 전 동결건조
12. 프로모션 번들 SKU

이 순서가 좋은 이유:

- 주문량이 많은 안정 SKU부터 성공률을 빨리 올릴 수 있다.
- 예외가 많은 상품은 뒤로 미뤄도 전체 매핑률이 크게 떨어지지 않는다.
- 운영 검수 포인트를 단계적으로 줄일 수 있다.

## 13. 향후 쿠팡, 카카오 확장 시에도 유지할 원칙

이 문서는 네이버 1차 구현 기준 문서지만, 향후 쿠팡과 카카오가 붙어도 핵심 원칙은 바뀌지 않는다.

그 원칙은 아래다.

- 외부 식별자는 채널마다 다를 수 있다.
- 하지만 매핑 순서는 같다.
- 먼저 외부 상품식별자로 상품군을 찾고
- 필요하면 옵션을 보고
- 필요하면 상품명을 보고
- 최종적으로는 내부 canonical 상품명으로 정규화한다.

즉 이 문서에서 설명한 방식은 네이버 전용 꼼수가 아니라, 멀티 커머스 구조의 기본 규칙으로 가져갈 수 있다.

## 14. 최종 결론

v2에서 상품 매핑은 아래처럼 이해하면 된다.

- 상품번호가 들어오면 먼저 그 상품번호가 어떤 상품군인지 찾는다.
- 일반 상품은 상품번호만으로 바로 연결한다.
- 옵션형 상품은 상품번호 뒤에 옵션까지 같이 본다.
- 애착트릿처럼 예외가 있는 상품은 상품명까지 다시 읽는다.
- 그래도 애매하면 기존 정규화 로직을 fallback으로 사용한다.
- 마지막까지 애매하면 사람이 검수한다.

한 줄로 다시 정리하면:

- `네이버 상품번호를 중심으로 매핑하되, 옵션과 상품명은 예외를 안전하게 처리하기 위한 보조 판단 수단으로 사용한다.`
